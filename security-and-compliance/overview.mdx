## Data Encryption

### Encryption at Rest

All data stored by Qovery is encrypted at rest using industry-standard encryption:

<Tabs>
  <Tab title="Databases">
    **Managed Databases**:
    - AES-256 encryption for all database storage
    - Automatic encryption enabled by default
    - Cloud provider managed keys (AWS KMS, Google Cloud KMS, Azure Key Vault)
    - Support for customer-managed encryption keys (CMEK)

    **Database Examples**:
    - PostgreSQL, MySQL: Encrypted tablespaces
    - MongoDB: Encrypted storage engine
    - Redis: Encrypted snapshots and AOF files
  </Tab>

  <Tab title="Object Storage">
    **Application Data**:
    - S3/GCS/Azure Blob: Server-side encryption (SSE)
    - AES-256 encryption standard
    - Encrypted bucket policies enforced
    - Versioning with encrypted objects

    **Build Artifacts**:
    - Container images encrypted in registry
    - Build cache encrypted at rest
    - Temporary files on encrypted volumes
  </Tab>

  <Tab title="Secrets">
    **Secret Management**:
    - Encrypted with AES-256-GCM
    - Stored in dedicated secret stores
    - Envelope encryption (data encrypted with DEK, DEK encrypted with KEK)
    - Automatic key rotation support

    **Supported Secret Stores**:
    - Qovery built-in secret store
    - AWS Secrets Manager
    - HashiCorp Vault
    - Doppler
    - External Secrets Operator
  </Tab>

  <Tab title="Volumes">
    **Persistent Storage**:
    - EBS volumes encrypted (AWS)
    - Persistent Disks encrypted (GCP)
    - Azure Disks encrypted (Azure)
    - Block Storage encrypted (Scaleway)

    **Encryption at Volume Level**:
    - Automatic encryption for all PersistentVolumeClaims
    - CSI driver managed encryption
    - No performance impact
  </Tab>
</Tabs>

### Encryption in Transit

All data transmitted through Qovery is encrypted using TLS 1.2 or higher:

<AccordionGroup>
  <Accordion title="Application Traffic" icon="globe">
    **HTTPS/TLS**:
    - Automatic SSL/TLS certificate provisioning via Let's Encrypt
    - TLS 1.2+ enforced (TLS 1.0/1.1 disabled)
    - Strong cipher suites only
    - HSTS enabled by default
    - Automatic certificate renewal

    **Custom Certificates**:
    - Upload your own SSL certificates
    - Wildcard certificate support
    - Multiple domains per certificate
    - SNI (Server Name Indication) support
  </Accordion>

  <Accordion title="API Communication" icon="code">
    **Qovery API**:
    - TLS 1.3 support
    - Certificate pinning
    - API token authentication
    - OAuth 2.0 / OIDC integration
    - Rate limiting and DDoS protection

    **Agent Communication**:
    - Mutual TLS (mTLS) between control plane and agents
    - Certificate-based authentication
    - Encrypted WebSocket connections
    - Automatic certificate rotation
  </Accordion>

  <Accordion title="Database Connections" icon="database">
    **Connection Encryption**:
    - SSL/TLS enforced for all database connections
    - Certificate verification enabled
    - Encrypted connection strings
    - No plaintext credentials

    **Internal Communication**:
    - Pod-to-pod encryption via service mesh (optional)
    - Network policies enforcing encrypted traffic
    - Private endpoints for managed services
  </Accordion>

  <Accordion title="Git Integration" icon="code-branch">
    **Repository Access**:
    - SSH keys with 4096-bit RSA or Ed25519
    - HTTPS with personal access tokens
    - OAuth tokens encrypted at rest
    - Webhook secrets validated
  </Accordion>
</AccordionGroup>

## Backup and Restore

Qovery provides comprehensive backup and disaster recovery capabilities:

### Database Backups

<Tabs>
  <Tab title="Automatic Backups">
    **Managed Database Backups**:
    - Automatic daily backups
    - Configurable backup schedule (hourly, daily, weekly)
    - Point-in-time recovery (PITR) available
    - Backup retention: 7 to 35 days (configurable)

    **Backup Process**:
    - Snapshot-based backups (no downtime)
    - Incremental backups for efficiency
    - Encrypted backups (AES-256)
    - Multi-region backup replication available
  </Tab>

  <Tab title="Manual Backups">
    **On-Demand Backups**:
    - Create backup before major changes
    - Export to external storage (S3, GCS, Azure Blob)
    - Backup tagging and labeling
    - Backup verification and testing

    **Backup via CLI**:
    ```bash
    # Create manual backup
    qovery database backup create --database-id <db-id>

    # List backups
    qovery database backup list --database-id <db-id>

    # Download backup
    qovery database backup download --backup-id <backup-id>
    ```
  </Tab>

  <Tab title="Restore Options">
    **Point-in-Time Recovery**:
    - Restore to any point within retention window
    - Transaction-level recovery granularity
    - Create new database from backup
    - Restore to different environment

    **Restore Process**:
    1. Select backup or point in time
    2. Choose target database (new or existing)
    3. Verify restored data
    4. Cutover when ready

    **Recovery Time Objective (RTO)**:
    - Typically 15-30 minutes for small databases (less than 100GB)
    - Larger databases may take longer
  </Tab>

  <Tab title="Disaster Recovery">
    **Multi-Region DR**:
    - Cross-region backup replication
    - Automated failover capabilities (enterprise)
    - Regular DR testing and validation
    - Runbook automation

    **Recovery Point Objective (RPO)**:
    - Continuous: < 1 minute (enterprise)
    - Standard: 24 hours
    - Configurable based on requirements
  </Tab>
</Tabs>

### Configuration Backups

**Infrastructure State**:
- Automatic backup of cluster configurations
- Environment configurations versioned
- Terraform state backup (if using Terraform)
- Helm releases tracked

**Backup Locations**:
- Cloud provider blob storage
- Encrypted and geo-replicated
- Immutable backups (WORM storage available)

## Security Best Practices

### For Developers

<Steps>
  <Step title="Use Secret Management">
    Never hardcode secrets in code or containers - use Qovery secret manager or external providers
  </Step>

  <Step title="Enable HTTPS">
    Always use HTTPS for public applications with automatic SSL certificates
  </Step>

  <Step title="Implement Least Privilege">
    Grant minimum required permissions to users and service accounts
  </Step>

  <Step title="Regular Updates">
    Keep dependencies updated and rebuild containers regularly
  </Step>

  <Step title="Enable MFA">
    Require multi-factor authentication for all team members
  </Step>
</Steps>

### For Operations

<Steps>
  <Step title="Network Segmentation">
    Use network policies to isolate workloads and environments
  </Step>

  <Step title="Monitor Security Events">
    Set up alerts for suspicious activities and failed authentications
  </Step>

  <Step title="Regular Backups">
    Test backup and restore procedures quarterly
  </Step>

  <Step title="Patch Management">
    Keep clusters and nodes updated with latest security patches
  </Step>

  <Step title="Security Scanning">
    Scan container images for vulnerabilities before deployment
  </Step>
</Steps>

## Incident Response

**Security Incident Process**:
1. Detection: Automated monitoring and alerting
2. Analysis: Security team investigates
3. Containment: Isolate affected resources
4. Eradication: Remove threat
5. Recovery: Restore normal operations
6. Post-Incident: Review and improve

**Contact for Security Issues**:
- Email: security@qovery.com
- For vulnerabilities: [Responsible Disclosure Policy](https://www.qovery.com/security)
