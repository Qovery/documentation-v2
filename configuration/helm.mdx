<img src="/images/logos/helm-icon.svg" alt="Helm" class="helm-icon" /> Helm is a package manager for Kubernetes that allows you to deploy pre-configured applications and services. Qovery provides native support for deploying Helm charts within your environment, making it easy to deploy complex applications with customizable configurations.

## Overview

Helm charts on Qovery are treated as a service type within your environment. You can deploy charts from:
- **Git Repository**: Store your Helm chart in a Git repository with full version control
- **Helm Repository**: Use public or private Helm repositories to deploy charts

This flexibility allows you to leverage the extensive Helm ecosystem while benefiting from Qovery's deployment management, monitoring, and logging capabilities.

## Creating a Helm Service

<Steps>
  <Step title="Navigate to Your Environment">
    Go to your project and select the environment where you want to deploy the Helm chart
  </Step>

  <Step title="Add Helm Service">
    Click **Create** and select **Helm** from the service types
  </Step>

  <Step title="Choose Source Type">
    Select your deployment source:
    - **Git Repository**: Deploy from a Git repository containing your Helm chart
    - **Helm Repository**: Deploy from a Helm chart repository
  </Step>

  <Step title="Configure and Deploy">
    Complete the configuration based on your selected source type (detailed below) and click **Create**
  </Step>
</Steps>

## Configuration Options

<Tabs>
  <Tab title="Git Repository">
    ### Deploy from Git Repository

    When deploying a Helm chart from a Git repository, configure the following:

    **Source Settings**:
    - **Git Provider**: Select your Git provider (GitHub, GitLab, Bitbucket, etc.)
    - **Repository**: Choose the repository containing your Helm chart
    - **Branch**: Select the branch to deploy from (e.g., `main`, `develop`)
    - **Root Helm Path**: Specify the path to the Helm chart directory within your repository

    **Example Repository Structure**:
    ```
    my-repo/
    ├── charts/
    │   └── my-app/
    │       ├── Chart.yaml
    │       ├── values.yaml
    │       └── templates/
    │           └── deployment.yaml
    └── README.md
    ```

    If your chart is in `charts/my-app`, set **Root Helm Path** to `charts/my-app`.

    <Info>
    Your repository must contain a valid Helm chart structure with `Chart.yaml` and a `templates/` directory.
    </Info>
  </Tab>

  <Tab title="Helm Repository">
    ### Deploy from Helm Repository

    When deploying from a Helm repository, configure:

    **Repository Settings**:
    - **Helm Repository**: Select from configured repositories or add a new one
    - **Chart Name**: The name of the chart (e.g., `nginx`, `postgresql`)
    - **Chart Version**: Specific version to deploy (e.g., `1.2.3`)
    - **Timeout**: Maximum time for deployment (default: 300 seconds)
    - **Arguments**: Additional Helm arguments (optional)

    **Popular Helm Repositories**:
    - Bitnami: `https://charts.bitnami.com/bitnami`
    - Stable: `https://charts.helm.sh/stable`
    - Prometheus: `https://prometheus-community.github.io/helm-charts`
    - Nginx: `https://kubernetes.github.io/ingress-nginx`

    <Tip>
    Using specific chart versions instead of "latest" ensures predictable deployments and easier rollbacks.
    </Tip>
  </Tab>
</Tabs>

## Overriding Values

Helm charts use a `values.yaml` file for configuration. Qovery provides three ways to override these values:

<AccordionGroup>
  <Accordion title="Values File Override">
    Upload or specify a custom `values.yaml` file to override default chart values.

    **Example values.yaml**:
    ```yaml
    replicaCount: 3

    image:
      repository: nginx
      tag: "1.21"
      pullPolicy: IfNotPresent

    service:
      type: LoadBalancer
      port: 80

    resources:
      limits:
        cpu: 500m
        memory: 512Mi
      requests:
        cpu: 250m
        memory: 256Mi
    ```

    This method is ideal for complex configurations with many overrides.
  </Accordion>

  <Accordion title="Raw YAML Override">
    Provide raw YAML directly in the Qovery console for quick overrides:

    ```yaml
    replicaCount: 2
    service:
      type: ClusterIP
    ```

    This method is perfect for environment-specific configurations.
  </Accordion>

  <Accordion title="Arguments Override">
    Use Helm command-line arguments for individual value overrides:

    ```bash
    --set replicaCount=3
    --set service.type=LoadBalancer
    --set image.tag=1.21
    ```

    This method works well for simple, single-value overrides.
  </Accordion>
</AccordionGroup>

### Using Environment Variables in Values

Qovery provides a special macro syntax to inject environment variables into your Helm values:

```yaml
database:
  host: qovery.env.DATABASE_HOST
  port: qovery.env.DATABASE_PORT
  username: qovery.env.DATABASE_USER
  password: qovery.env.DATABASE_PASSWORD

api:
  key: qovery.env.API_KEY
  url: qovery.env.API_URL
```

**Syntax**: `qovery.env.<ENV_VAR_NAME>`

This allows you to:
- Use Qovery-managed secrets
- Reference other services in your environment
- Keep sensitive data out of your chart configuration

<Warning>
Environment variables referenced with `qovery.env.*` must exist in your Helm service configuration, otherwise deployment will fail.
</Warning>

## Network Configuration

Configure how your Helm chart exposes services and handles network traffic:

### Kubernetes Service Configuration

Your Helm chart should define Kubernetes services. Qovery automatically handles:
- Service discovery within the environment
- Internal DNS resolution
- Network policies

### Port Exposure

Configure which ports to expose:

**Port Configuration**:
- **Service Port**: The port your Kubernetes service exposes
- **Protocol**: HTTP, HTTPS, TCP, or gRPC
- **Public Access**: Toggle to expose via internet

**Example Configuration**:
```yaml
Web Application:
  Service Port: 80
  Protocol: HTTPS
  Public Access: Enabled

API Service:
  Service Port: 8080
  Protocol: gRPC
  Public Access: Enabled

Internal Service:
  Service Port: 9000
  Protocol: HTTP
  Public Access: Disabled
```

## Custom Domains

### Automatic Domain Generation

When you enable public access, Qovery automatically generates a domain for your Helm service:

**Domain Pattern**:
```
p<PORT>-<RANDOM_ID>-<SERVICE_ID>-gtw.<CLUSTER_ID>.bool.sh
```

**Example**:
```
p80-zdf72de72-z709e1a88-gtw.za8ad0657.bool.sh
```

Breaking down the pattern:
- `p80`: Port 80 is exposed
- `zdf72de72`: Random identifier
- `z709e1a88`: Service ID
- `gtw`: Gateway indicator
- `za8ad0657`: Cluster ID
- `bool.sh`: Qovery domain suffix

This domain is automatically:
- SSL/TLS secured with Let's Encrypt
- Load balanced
- Health checked

### Using Custom Domains

Add your own domain to your Helm service:

<img src="/images/configuration/application/custom-domain.png" alt="Custom Domain Configuration" />

<Steps>
  <Step title="Add Custom Domain">
    Navigate to your Helm service → **Settings** → **Domains** → **Add Domain**
  </Step>

  <Step title="Enter Domain Name">
    Enter your domain (e.g., `app.example.com` or `api.mycompany.io`)
  </Step>

  <Step title="Configure DNS">
    Add a CNAME record in your DNS provider pointing to the Qovery-generated domain:

    ```
    Type: CNAME
    Name: app
    Value: p80-zdf72de72-z709e1a88-gtw.za8ad0657.bool.sh
    TTL: 3600
    ```
  </Step>

  <Step title="Verify and Deploy">
    - Qovery automatically verifies DNS configuration
    - SSL/TLS certificate is provisioned via Let's Encrypt
    - Redeploy your Helm service to activate the custom domain
  </Step>
</Steps>

<img src="/images/configuration/application/domain_check.png" alt="Domain Validation Success" />

<Info>
Certificate provisioning typically takes 2-5 minutes. You can monitor the status in the Qovery console.
</Info>

### Multiple Custom Domains

You can add multiple custom domains to a single Helm service:

```
Primary: app.example.com
Aliases:
  - www.example.com
  - app.mycompany.io
  - api.example.com
```

All domains will be:
- Automatically SSL/TLS secured
- Load balanced to the same service
- Health checked independently

## Deployment Management

### Deployment Actions

Manage your Helm service deployments:

**Available Actions**:
- **Deploy**: Deploy new changes or updates
- **Redeploy**: Redeploy current version
- **Stop**: Stop the Helm release (pods removed)
- **Restart**: Restart all pods in the release

### Viewing Logs

Access logs for your Helm service:

<Steps>
  <Step title="Navigate to Logs">
    Click on your Helm service → **Logs** tab
  </Step>

  <Step title="View Logs">
    - **Real-time logs**: Live tail of application logs
    - **Historical logs**: Search past logs by time range
    - **Pod selection**: View logs from specific pods
    - **Log filtering**: Filter by log level or search terms
  </Step>
</Steps>

### Deployment History

Track all deployments and changes:

- View deployment timeline
- See who triggered each deployment
- Access deployment logs and status
- Rollback to previous versions

Duplicate a Helm service with its configuration:

## Advanced Settings

Configure advanced Helm deployment options:

<AccordionGroup>
  <Accordion title="Timeout Configuration">
    Set maximum time for Helm operations:

    ```yaml
    Install Timeout: 300s
    Upgrade Timeout: 600s
    Rollback Timeout: 300s
    Delete Timeout: 300s
    ```

    Increase timeouts for charts with:
    - Long initialization times
    - Large number of resources
    - External dependency checks
  </Accordion>

  <Accordion title="Helm Arguments">
    Pass additional arguments to Helm commands:

    **Common Arguments**:
    ```bash
    --atomic                    # Rollback on failure
    --wait                      # Wait for resources to be ready
    --wait-for-jobs            # Wait for jobs to complete
    --create-namespace         # Create namespace if missing
    --dependency-update        # Update dependencies before install
    --skip-crds               # Skip CRD installation
    ```

    **Example**:
    ```bash
    --atomic --wait --wait-for-jobs
    ```
  </Accordion>

  <Accordion title="Hook Management">
    Control Helm hook behavior:

    - **Before Install**: Run jobs before chart installation
    - **After Install**: Run jobs after chart installation
    - **Before Upgrade**: Run jobs before chart upgrade
    - **After Upgrade**: Run jobs after chart upgrade
    - **Before Delete**: Run jobs before chart deletion
    - **After Delete**: Run jobs after chart deletion

    Hooks are defined in your Helm chart templates.
  </Accordion>

  <Accordion title="Resource Tracking">
    Monitor resources created by your Helm chart:

    - Deployments and StatefulSets
    - Services and Ingresses
    - ConfigMaps and Secrets
    - Persistent Volume Claims
    - Jobs and CronJobs

    View all resources in the Qovery console under **Resources** tab.
  </Accordion>
</AccordionGroup>

## Deleting a Helm Service

Remove a Helm service from your environment:

<Steps>
  <Step title="Stop Service">
    First, stop the service to ensure graceful shutdown:
    - Click the **3 dots menu** → **Stop**
    - Wait for all pods to terminate
  </Step>

  <Step title="Delete Service">
    Click the **3 dots menu** → **Remove helm**
  </Step>

  <Step title="Confirm Deletion">
    - Review what will be deleted
    - Confirm by typing the service name
    - Click **Delete** to remove the service
  </Step>
</Steps>

<Warning>
Deleting a Helm service will:
- Remove all Kubernetes resources created by the chart
- Delete all associated logs and metrics
- Remove custom domains and DNS configurations
- This action cannot be undone

Ensure you have backups of any important data before deletion.
</Warning>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Deployment Fails">
    **Common Causes**:
    - Invalid values configuration
    - Missing dependencies
    - Insufficient resources
    - Timeout too short

    **Solutions**:
    - Check deployment logs for errors
    - Validate values.yaml syntax
    - Ensure all environment variables exist
    - Increase timeout if needed
    - Verify Helm chart compatibility
  </Accordion>

  <Accordion title="Service Not Accessible">
    **Check**:
    - Port configuration matches chart service
    - Public access is enabled (if needed)
    - Custom domain DNS is configured correctly
    - SSL certificate has been provisioned
    - Pod health checks are passing

    **Debug Steps**:
    1. Check pod status and logs
    2. Verify service is running: `kubectl get svc`
    3. Test internal connectivity first
    4. Verify external DNS and SSL
  </Accordion>

  <Accordion title="Value Overrides Not Applied">
    **Verify**:
    - Values file syntax is correct (valid YAML)
    - Override precedence: Arguments > Raw YAML > Values File
    - Chart supports the values you're overriding
    - Values are in the correct nested structure

    **Test Locally**:
    ```bash
    helm template my-release ./chart -f values.yaml
    ```
    This shows the final rendered templates.
  </Accordion>

  <Accordion title="Environment Variables Not Resolved">
    **Check**:
    - Environment variables exist in Helm service config
    - Syntax is correct: `qovery.env.VAR_NAME`
    - Variables are set before deployment
    - No typos in variable names

    **Example**:
    ```yaml
    # Correct
    database:
      host: qovery.env.DB_HOST

    # Incorrect
    database:
      host: $DB_HOST              # Wrong syntax
      host: qovery.env.db_host    # Case sensitive
    ```
  </Accordion>
</AccordionGroup>
