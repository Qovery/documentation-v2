Clusters in Qovery are Kubernetes clusters that run your applications, databases, and services. Qovery supports multi-cloud deployments across AWS, GCP, Azure, and Scaleway, providing a unified interface to manage infrastructure regardless of the underlying cloud provider.

## Overview

A cluster is the foundational infrastructure where all your environments and services run. Qovery handles the complexity of Kubernetes cluster management, including:

- Automated cluster provisioning
- Node management and auto-scaling
- Kubernetes version upgrades
- Security patching and vulnerability management
- Load balancing and networking
- Monitoring and observability
- Cost optimization

**Supported Cloud Providers**:
- Amazon Web Services (AWS)
- Google Cloud Platform (GCP)
- Microsoft Azure
- Scaleway

## Creating a Cluster

<Steps>
  <Step title="Add Cluster">
    Navigate to your organization settings and click **Add Cluster**
  </Step>

  <Step title="Choose Cloud Provider">
    Select your cloud provider:
    - **AWS**: Amazon Elastic Kubernetes Service (EKS)
    - **GCP**: Google Kubernetes Engine (GKE)
    - **Azure**: Azure Kubernetes Service (AKS)
    - **Scaleway**: Scaleway Kubernetes Kapsule
  </Step>

  <Step title="Configure Credentials">
    Provide cloud provider credentials:
    - AWS: IAM access keys or role ARN
    - GCP: Service account key
    - Azure: Service principal credentials
    - Scaleway: API keys
  </Step>

  <Step title="Configure Cluster Settings">
    - **Cluster Name**: Unique identifier
    - **Region**: Cloud provider region
    - **Kubernetes Version**: K8s version to deploy
    - **Node Configuration**: Instance types and sizes
    - **Auto-scaling**: Enable/disable node auto-scaling
  </Step>

  <Step title="Review and Create">
    Review your configuration and click **Create Cluster**. Provisioning typically takes 15-30 minutes.
  </Step>
</Steps>

<Info>
Cluster creation is a one-time process. Once created, Qovery manages the cluster lifecycle, including updates and maintenance.
</Info>

### Stop Cluster

Temporarily stop a cluster to save costs:

<Steps>
  <Step title="Stop Cluster">
    Click on cluster ‚Üí **Actions** ‚Üí **Stop**
  </Step>

  <Step title="Confirm Action">
    Confirm you want to stop the cluster. All services will be stopped.
  </Step>

  <Step title="Wait for Completion">
    Cluster will be stopped. Infrastructure is preserved but not running.
  </Step>
</Steps>

**What Happens**:
- All applications and services are stopped
- Nodes are terminated
- Persistent data is preserved
- Cloud resources are deallocated (costs reduced to storage only)

**To Restart**:
- Click **Restart** action
- Cluster will be reprovisioned
- Services will be redeployed automatically

<Tip>
Stop clusters during off-hours for development/testing environments to significantly reduce costs.
</Tip>

### Restart Cluster

Restart a stopped cluster:

<Steps>
  <Step title="Restart Cluster">
    Click on cluster ‚Üí **Actions** ‚Üí **Restart**
  </Step>

  <Step title="Wait for Provisioning">
    Cluster will be reprovisioned (15-30 minutes)
  </Step>

  <Step title="Services Redeploy">
    All services will automatically redeploy once cluster is ready
  </Step>
</Steps>

### Delete Cluster

Permanently remove a cluster:

<Warning>
Deleting a cluster is permanent and cannot be undone. All services, data, and configurations will be lost unless backed up.
</Warning>

<Steps>
  <Step title="Stop Services">
    Stop all environments and services first (recommended)
  </Step>

  <Step title="Initiate Deletion">
    Click on cluster ‚Üí **Actions** ‚Üí **Delete**
  </Step>

  <Step title="Choose Deletion Mode">
    Select deletion mode:
    - **Delete cloud resources + configuration**: Removes everything (recommended)
    - **Delete configuration only**: Keeps cloud resources (for manual management)
  </Step>

  <Step title="Confirm Deletion">
    Type the cluster name to confirm deletion
  </Step>

  <Step title="Monitor Progress">
    Deletion process will:
    1. Stop all services
    2. Delete Kubernetes resources
    3. Terminate nodes
    4. Remove load balancers and networking
    5. Delete persistent volumes (if selected)
    6. Clean up cloud provider resources
  </Step>
</Steps>

**Deletion Options**:

**Option 1: Delete Cloud Resources + Configuration** (Recommended)
- Removes all cloud resources
- Deletes Qovery configuration
- Complete cleanup
- No ongoing costs

**Option 2: Delete Configuration Only**
- Removes Qovery management
- Keeps cloud resources running
- Continue managing manually
- Ongoing cloud costs

<Tip>
Before deleting, export any important configuration using Terraform or Qovery CLI for potential future recreation.
</Tip>

### Access Kubeconfig

Download Kubernetes configuration for direct cluster access:

<Steps>
  <Step title="Download Kubeconfig">
    Click on cluster ‚Üí **Actions** ‚Üí **Get Kubeconfig**
  </Step>

  <Step title="Save Configuration">
    Save the kubeconfig file to `~/.kube/config` or a custom location
  </Step>

  <Step title="Use with kubectl">
    ```bash
    # Using default location
    kubectl get pods --all-namespaces

    # Using custom location
    kubectl --kubeconfig=/path/to/config get pods
    ```
  </Step>
</Steps>

**Security Considerations**:
- Kubeconfig provides cluster-admin access
- Store securely, never commit to Git
- Rotate credentials periodically
- Use RBAC for team access instead

<Warning>
Direct cluster access should be used sparingly. Qovery provides safer interfaces for most operations. Use kubectl only for debugging or advanced operations.
</Warning>

## Custom Domain with Cloudflare (Beta)

Configure wildcard TLS certificates using Cloudflare:

### Prerequisites

- Cloudflare account
- Domain managed by Cloudflare
- Cloudflare API token

### Configuration Steps

<Steps>
  <Step title="Enable Custom Domain">
    Go to cluster ‚Üí **Settings** ‚Üí **Custom Domain**
  </Step>

  <Step title="Configure Cloudflare">
    - Enter your domain (e.g., `*.myapp.com`)
    - Provide Cloudflare API token
    - Select Cloudflare zone
  </Step>

  <Step title="Verify DNS">
    Qovery will create required DNS records:
    - `*.myapp.com` pointing to cluster load balancer
    - TXT records for certificate validation
  </Step>

  <Step title="Certificate Provisioning">
    - Wildcard TLS certificate is automatically provisioned
    - Certificate auto-renewal configured
    - All applications can use custom subdomains
  </Step>
</Steps>

**Benefits**:
- Single wildcard certificate for all services
- Automatic DNS management
- Auto-renewal
- Better SEO and branding

**Example Usage**:
```
Before: app-zdf72de72.za8ad0657.bool.sh
After:  api.myapp.com, admin.myapp.com, app.myapp.com
```

<Info>
Custom domain feature is in Beta. Additional DNS providers will be supported in future releases.
</Info>

## Kubernetes Version Upgrades

Qovery manages Kubernetes version upgrades to ensure security and access to latest features:

### Upgrade Process

**Automatic Notifications**:
- Email notifications when upgrades available
- In-console banners for recommended upgrades
- Warnings for versions nearing end-of-support

**Upgrade Methods**:

<Tabs>
  <Tab title="Managed Upgrades (Recommended)">
    Qovery handles the entire upgrade process:

    1. **Schedule Upgrade**: Choose upgrade window
    2. **Pre-upgrade Checks**: Qovery validates cluster health
    3. **Rolling Upgrade**: Nodes upgraded one at a time
    4. **Service Health Checks**: Ensures services remain healthy
    5. **Rollback Ready**: Automatic rollback if issues detected

    **Timeline**: 30-60 minutes depending on cluster size
  </Tab>

  <Tab title="Manual Upgrades (Not Recommended)">
    <Warning>
    Manual Kubernetes upgrades can cause service disruptions and compatibility issues. Always use Qovery-managed upgrades.
    </Warning>

    If you must upgrade manually:
    - Coordinate with Qovery support
    - Ensure all services are compatible
    - Test in non-production first
    - Have rollback plan ready
  </Tab>
</Tabs>

### Version Support Lifecycle

**Qovery Support Policy**:
- Latest 3 Kubernetes minor versions supported
- Security patches applied automatically
- 3-month notice before version end-of-support
- Forced upgrades only after extended grace period

**Example Timeline**:
```
Current Version: 1.28
- 1.28: Fully supported ‚úÖ
- 1.27: Fully supported ‚úÖ
- 1.26: Supported ‚úÖ
- 1.25: Deprecated (upgrade recommended) ‚ö†Ô∏è
- 1.24: End of support (forced upgrade soon) üî¥
```

### Best Practices

## Troubleshooting

<AccordionGroup>
  <Accordion title="Cluster Shows Warning Status">
    **Possible Causes**:
    - One or more nodes unhealthy
    - High resource utilization
    - Network connectivity issues
    - Recent failed deployments

    **Actions**:
    1. Check cluster overview for specific warnings
    2. Review node status and restart unhealthy nodes
    3. Check resource utilization and scale if needed
    4. Review recent deployment logs
    5. Contact support if issue persists
  </Accordion>

  <Accordion title="Cluster Deployment Failed">
    **Common Issues**:
    - Insufficient cloud provider quotas
    - Invalid credentials
    - Region or instance type unavailable
    - Network configuration errors

    **Resolution**:
    1. Check deployment logs for specific error
    2. Verify cloud provider quotas and limits
    3. Confirm credentials are valid and have required permissions
    4. Try alternative region or instance type
    5. Contact Qovery support with deployment ID
  </Accordion>

  <Accordion title="Services Won't Deploy to Cluster">
    **Check**:
    - Cluster status is Running (‚úÖ)
    - Sufficient resources available
    - No node affinity conflicts
    - Network policies allow traffic
    - Image pull secrets configured

    **Debug**:
    1. View service deployment logs
    2. Check pod events: `kubectl describe pod <pod-name>`
    3. Verify resource requests don't exceed capacity
    4. Check cluster networking configuration
    5. Ensure container registry credentials are valid
  </Accordion>

  <Accordion title="High Costs">
    **Optimize**:
    - Review node instance types (downsize if possible)
    - Reduce number of nodes
    - Enable auto-scaling to scale down during low usage
    - Use spot/preemptible instances for non-critical workloads
    - Stop dev/test clusters when not needed
    - Review application resource requests (may be overallocated)
    - Delete unused persistent volumes

    **Cost Analysis**:
    - View cost breakdown in cloud provider console
    - Use Qovery resource optimization recommendations
    - Monitor trends over time
  </Accordion>

  <Accordion title="Cannot Access Cluster with kubectl">
    **Troubleshooting**:
    1. Verify kubeconfig is correctly downloaded and saved
    2. Check kubectl is installed: `kubectl version --client`
    3. Test connection: `kubectl cluster-info`
    4. Verify your IP isn't blocked by firewall
    5. Confirm kubeconfig hasn't expired
    6. Try downloading fresh kubeconfig from Qovery

    **Common Errors**:
    ```bash
    # Error: Unable to connect to the server
    # Solution: Check network connectivity and kubeconfig

    # Error: Unauthorized
    # Solution: Download fresh kubeconfig from Qovery

    # Error: Certificate expired
    # Solution: Cluster certificates may need rotation
    ```
  </Accordion>
</AccordionGroup>
