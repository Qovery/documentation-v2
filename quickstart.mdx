## Detailed Use Case Comparison

<AccordionGroup>
  <Accordion title="Testing & Learning" icon="flask">
    **Use Case: Local Development & Exploration**

    **Best for:**
    - Developers new to Qovery wanting to explore features
    - Teams evaluating Qovery for adoption
    - Students and learners practicing DevOps
    - Building POCs before cloud deployment

    **What you can do:**
    - Deploy apps, databases, and jobs locally
    - Test GitOps workflows with GitHub/GitLab
    - Practice Qovery CLI and Console
    - Experiment with environment configurations
    - Learn Kubernetes concepts without cloud costs

    **Limitations:**
    - Runs on laptop (limited resources)
    - Not for production workloads
    - Single-node cluster

    **Path:** Docker Desktop | **Time:** 15 min | **Cost:** Free
  </Accordion>

  <Accordion title="Production Applications" icon="rocket">
    **Use Case: SaaS, Web Apps, APIs, Microservices**

    **Best for:**
    - SaaS platforms with paying customers
    - High-traffic web applications
    - RESTful/GraphQL/gRPC APIs
    - Microservices architectures
    - Production workloads requiring 99.9% uptime

    **What you can do:**
    - Deploy Node.js, Python, Go, Java, PHP, Ruby, .NET apps
    - Auto-scale based on traffic (CPU/memory)
    - Managed databases with automatic backups
    - Custom domains with automatic SSL
    - Zero-downtime deployments
    - Monitor with Datadog, New Relic, or Qovery Observe

    **Infrastructure:**
    - Qovery provisions and manages Kubernetes
    - Multi-AZ for high availability
    - Automatic updates and security patches

    **Path:** Managed Cluster (AWS, GCP, Azure, Scaleway) | **Time:** 30 min
  </Accordion>

  <Accordion title="Preview & Ephemeral Environments" icon="code-branch">
    **Use Case: Temporary Environments for Testing & Review**

    **Best for:**
    - Preview environments for every pull request
    - QA teams testing features in isolation
    - Product managers reviewing features before release
    - Stakeholder demos without affecting production
    - Short-lived environments for testing

    **What you can do:**
    - Automatically create environment when PR is opened
    - Full-stack preview with application + database
    - Share unique URL with team for review
    - Test API integrations in isolated environment
    - Automatic cleanup when PR is closed/merged
    - Cost-effective (only pay when environments are active)

    **How it works:**
    1. Open pull request in GitHub/GitLab
    2. Qovery auto-creates preview environment
    3. Get unique URL to access (e.g., `pr-123.preview.example.com`)
    4. Test, review, and iterate
    5. Merge PR → Environment auto-deleted

    **Path:** Managed Cluster (AWS, GCP, Azure, Scaleway) | **Time:** 30 min setup
  </Accordion>

  <Accordion title="Enterprise & Custom Requirements" icon="building">
    **Use Case: Existing Clusters, Compliance, On-Premise**

    **Best for:**
    - Organizations with existing Kubernetes clusters
    - Regulated industries (Healthcare, Finance, Government)
    - Companies with strict compliance (HIPAA, PCI-DSS, SOC2)
    - On-premise or air-gapped environments
    - Hybrid cloud architectures
    - Teams with dedicated platform/SRE engineers

    **What you can do:**
    - Connect any Kubernetes 1.24+ cluster to Qovery
    - Maintain complete control over networking, security, storage
    - Custom Kubernetes configurations and policies
    - Use with AWS, GCP, Azure, DigitalOcean, OVH, Civo, on-premise
    - Implement custom monitoring and logging
    - Define your own autoscaling rules

    **You manage:**
    - Kubernetes cluster provisioning and upgrades
    - Infrastructure security and networking
    - Backup and disaster recovery

    **Qovery provides:**
    - Developer-friendly UI and CLI
    - GitOps workflows and CI/CD
    - Environment management
    - Application deployment and monitoring

    **Path:** Bring Your Own Cluster (BYOK) | **Time:** 20 min | **Control:** 100%
  </Accordion>
</AccordionGroup>
