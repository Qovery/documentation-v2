## Overview

Qovery Observe is Qovery's built-in observability solution that provides logs, metrics, and events for all your applications without any additional configuration. Built on industry-standard open-source tools (Loki, Prometheus, Grafana), it offers immediate insights into your applications' health and performance.

<Accordion title="Application Metrics" icon="chart-area">
  **HTTP Metrics**:
  - Request rate (req/sec)
  - Response times (p50, p95, p99)
  - Status code distribution
  - Error rates

  **Custom Metrics**:
  Expose Prometheus metrics from your application:

  ```javascript
  // Node.js example with prom-client
  const client = require('prom-client');
  const register = new client.Registry();

  const httpRequestDuration = new client.Histogram({
    name: 'http_request_duration_seconds',
    help: 'Duration of HTTP requests in seconds',
    labelNames: ['method', 'route', 'status_code']
  });

  register.registerMetric(httpRequestDuration);

  // Expose metrics endpoint
  app.get('/metrics', (req, res) => {
    res.set('Content-Type', register.contentType);
    res.end(register.metrics());
  });
  ```

  ```python
  # Python example with prometheus_client
  from prometheus_client import Histogram, generate_latest
  from flask import Flask, Response

  app = Flask(__name__)

  REQUEST_DURATION = Histogram(
      'http_request_duration_seconds',
      'Duration of HTTP requests in seconds',
      ['method', 'endpoint', 'status']
  )

  @app.route('/metrics')
  def metrics():
      return Response(generate_latest(), mimetype='text/plain')
  ```

  **Configure Metrics Port**:
  In your application settings, add metrics port:
  - Port: `9090` (or your metrics port)
  - Protocol: HTTP
  - Service type: Internal (not publicly exposed)
</Accordion>

### Events

<Accordion title="Kubernetes Events" icon="cubes">
  **Event Types**:

  **Pod Lifecycle**:
  - `Scheduled`: Pod assigned to node
  - `Pulling`: Pulling container image
  - `Pulled`: Image pulled successfully
  - `Created`: Container created
  - `Started`: Container started
  - `Killing`: Container being stopped
  - `FailedScheduling`: Cannot schedule pod
  - `FailedMount`: Volume mount failed

  **Resource Events**:
  - `ScalingReplicaSet`: Scaling replicas
  - `SuccessfulCreate`: Resource created
  - `SuccessfulDelete`: Resource deleted

  **Warning Events**:
  - `BackOff`: Container restart backoff
  - `Unhealthy`: Health check failed
  - `FailedKillPod`: Failed to kill pod
  - `OOMKilled`: Out of memory killed
</Accordion>

<Accordion title="Deployment Events" icon="rocket">
  **Qovery Deployment Tracking**:
  - **Started**: Deployment initiated
  - **Building**: Building container image
  - **Pushing**: Pushing to registry
  - **Deploying**: Rolling out to Kubernetes
  - **Running**: Deployment successful
  - **Failed**: Deployment failed (with reason)
  - **Rolled Back**: Automatic rollback triggered

  **Timeline View**:
  - Visual timeline of all deployments
  - Duration of each phase
  - Success/failure indicators
  - Links to logs for each phase
</Accordion>

<Accordion title="Health Check Events" icon="heartbeat">
  **Probe Failures**:
  - **Liveness probe failed**: Application not responding
  - **Readiness probe failed**: Not ready for traffic
  - **Startup probe failed**: Initial startup check failed

  **Event Details**:
  - Timestamp of failure
  - Number of consecutive failures
  - Probe configuration
  - Container restart count
  - Recovery actions taken

  **Alerts**:
  Configure alerts for health check failures:
  1. Go to application settings
  2. Navigate to Health Checks
  3. Enable health check monitoring
  4. Set up alert channels (email, Slack, webhook)
</Accordion>

## Best Practices

<CardGroup cols={2}>
  <Card title="Structured Logging" icon="code">
    **Use JSON format** for better parsing:
    ```json
    {
      "timestamp": "2025-01-15T10:30:00Z",
      "level": "error",
      "message": "Database connection failed",
      "service": "api",
      "trace_id": "abc123"
    }
    ```

    Benefits:
    - Better searchability
    - Easier filtering
    - Machine-readable
    - Correlation with traces
  </Card>

  <Card title="Log Levels" icon="layer-group">
    **Use appropriate log levels**:
    - `DEBUG`: Detailed debugging info (disable in production)
    - `INFO`: General informational messages
    - `WARN`: Warning messages (potential issues)
    - `ERROR`: Error messages (requires attention)
    - `FATAL`: Critical errors (service down)

    Avoid excessive INFO logs in production.
  </Card>

  <Card title="Custom Metrics" icon="chart-simple">
    **Expose application metrics**:
    - Business metrics (orders, signups)
    - Performance metrics (latency, throughput)
    - Error rates
    - Queue lengths

    Use Prometheus client libraries.
  </Card>

  <Card title="Health Checks" icon="heart-pulse">
    **Configure proper health checks**:
    - **Liveness**: Check if app is alive
    - **Readiness**: Check if ready for traffic
    - **Startup**: Initial startup check

    Return HTTP 200 for healthy, 500+ for unhealthy.
  </Card>
</CardGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="No Logs Appearing">
    **Problem**: Logs not showing in Qovery Console

    **Solutions**:
    1. **Check stdout/stderr**: Ensure application logs to stdout/stderr, not files
       ```javascript
       // Good
       console.log("Message");

       // Bad (won't appear in Qovery)
       fs.writeFileSync('/var/log/app.log', "Message");
       ```

    2. **Verify application is running**: Check pod status
    3. **Wait for aggregation**: Logs may take 10-30 seconds to appear
    4. **Check time range**: Ensure correct time range selected
    5. **Verify log format**: Very long log lines may be truncated
  </Accordion>

  <Accordion title="Metrics Not Showing">
    **Problem**: Custom metrics not appearing

    **Solutions**:
    1. **Check metrics endpoint**: Verify `/metrics` endpoint returns data
       ```bash
       curl http://localhost:9090/metrics
       ```

    2. **Configure port**: Ensure metrics port configured in Qovery
    3. **Check ServiceMonitor**: Verify Prometheus is scraping
    4. **Validate format**: Use Prometheus exposition format
    5. **Check labels**: Ensure consistent label names
  </Accordion>

  <Accordion title="High Memory Usage">
    **Problem**: Logs using too much memory

    **Solutions**:
    1. **Reduce log verbosity**: Disable DEBUG logs in production
    2. **Limit log size**: Truncate very large log messages
    3. **Enable log sampling**: Only log percentage of requests
    4. **Use external tool**: For extremely high log volumes, consider external tool
    5. **Rotate logs**: Ensure log rotation is working
  </Accordion>

  <Accordion title="Events Not Showing">
    **Problem**: Kubernetes events missing

    **Solutions**:
    1. **Check time range**: Events may be outside selected range
    2. **Verify event severity**: Check if filtering by severity
    3. **Wait for propagation**: Events may take 1-2 minutes
    4. **Check permissions**: Verify Qovery has cluster access
  </Accordion>
</AccordionGroup>
