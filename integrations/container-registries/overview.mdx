## Overview

Qovery supports all major container registries, allowing you to deploy pre-built container images or use them as build caches. Connect public or private registries to your Qovery organization and deploy images across your environments.

## Use Cases

### Deploy Pre-Built Images

Deploy container images that have been built by your CI/CD pipeline:

1. **CI/CD builds and pushes** image to registry
2. **Tag with commit SHA** or version number
3. **Qovery pulls image** and deploys to Kubernetes
4. **Zero build time** on Qovery - instant deployments

**Benefits**:
- Faster deployments (no build step)
- Consistent images across environments
- Better control over build process
- Separation of concerns (build vs deploy)

### Build Cache

Use registry as a cache to speed up builds:

- **Docker layer caching**: Reuse unchanged layers
- **Dependency caching**: Cache dependencies between builds
- **50% faster builds** on average

### Image Mirroring

Qovery automatically mirrors images for reliability:

- **High availability**: Images cached in Qovery's registry
- **Faster pulls**: Reduced latency for deployments
- **Dependency protection**: Deployments work even if source registry is down

## Configuration

### Adding a Container Registry

<Steps>
  <Step title="Navigate to Settings">
    Go to your **Organization Settings** → **Container Registries**
  </Step>

  <Step title="Add New Registry">
    Click **Add Container Registry** button
  </Step>

  <Step title="Select Registry Type">
    Choose your registry provider from the list
  </Step>

  <Step title="Provide Credentials">
    Enter registry URL and authentication details:
    - Registry name (for identification)
    - Registry URL
    - Username or Access Key
    - Password or Secret Key
  </Step>

  <Step title="Test Connection">
    Qovery will verify the credentials and connection
  </Step>

  <Step title="Save">
    Registry is now available for all projects in your organization
  </Step>
</Steps>

### Registry Configuration Options

Different registries require different authentication methods:

<Tabs>
  <Tab title="AWS ECR">
    **Authentication Options**:
    - **Access Keys**: AWS Access Key ID + Secret Access Key
    - **IAM Role** (recommended): For EKS clusters with IRSA

    **Required Permissions**:
    ```json
    {
      "Version": "2012-10-17",
      "Statement": [
        {
          "Effect": "Allow",
          "Action": [
            "ecr:GetAuthorizationToken",
            "ecr:BatchCheckLayerAvailability",
            "ecr:GetDownloadUrlForLayer",
            "ecr:BatchGetImage"
          ],
          "Resource": "*"
        }
      ]
    }
    ```

    **Registry URL Format**:
    ```
    https://<aws_account_id>.dkr.ecr.<region>.amazonaws.com
    ```
  </Tab>

  <Tab title="Docker Hub">
    **Authentication**:
    - Username: Your Docker Hub username
    - Password: Docker Hub password or access token

    **Registry URL**:
    ```
    https://index.docker.io/v1/
    ```

    **Pro Tips**:
    - Use access tokens instead of passwords for better security
    - Enable 2FA on Docker Hub account
    - Consider rate limits (100 pulls/6hrs for free tier)
  </Tab>

  <Tab title="GitHub CR">
    **Authentication**:
    - Username: Your GitHub username
    - Password: GitHub Personal Access Token

    **Required Token Scopes**:
    - `read:packages`
    - `write:packages` (if pushing images)

    **Registry URL**:
    ```
    https://ghcr.io
    ```

    **Image Format**:
    ```
    ghcr.io/<owner>/<image-name>:<tag>
    ```
  </Tab>

  <Tab title="Generic">
    **Works with any Docker-compatible registry**:
    - Harbor
    - Nexus
    - JFrog Artifactory
    - Self-hosted Docker registry
    - Quay.io

    **Configuration**:
    - Registry URL (with protocol: https://)
    - Username
    - Password or token
    - Optional: Skip TLS verification (not recommended)
  </Tab>
</Tabs>

## Deploying from Registry

Once a registry is connected, you can deploy images:

### Create Application from Registry

<Steps>
  <Step title="Create Application">
    In your environment, click **Create** → **Application**
  </Step>

  <Step title="Select Container Registry">
    Choose **Container Registry** as the source
  </Step>

  <Step title="Select Registry">
    Pick your configured registry from the dropdown
  </Step>

  <Step title="Specify Image">
    - **Image Name**: Full image path (e.g., `my-org/my-app`)
    - **Tag**: Image tag to deploy (e.g., `latest`, `v1.2.3`, `${COMMIT_SHA}`)
  </Step>

  <Step title="Configure Application">
    Set ports, environment variables, resources, etc.
  </Step>

  <Step title="Deploy">
    Click **Deploy** to launch your application
  </Step>
</Steps>

### Dynamic Tags

Use variables in image tags for dynamic deployments:

```yaml
Image: my-org/my-app
Tag: ${COMMIT_SHA}  # Automatically replaced with Git commit
```

**Available Variables**:
- `${COMMIT_SHA}`: Git commit SHA
- `${BRANCH_NAME}`: Git branch name
- `${TAG_NAME}`: Git tag name

## Registry Comparison

| Registry | Best For | Pros | Cons |
|----------|----------|------|------|
| **AWS ECR** | AWS deployments | Deep AWS integration, IAM roles, Regional | AWS-only, complex permissions |
| **Docker Hub** | Public images | Largest image library, easy to use | Rate limits, slower pulls |
| **GCR/Artifact Registry** | GCP deployments | GCP integration, fast, reliable | GCP-only |
| **GitHub CR** | GitHub projects | Integrated with GitHub, free for public | Tied to GitHub |
| **GitLab CR** | GitLab projects | Built-in CI/CD, integrated | Tied to GitLab |
| **Scaleway** | Scaleway deployments | European hosting, GDPR compliant | Smaller ecosystem |
| **Generic** | Private registries | Maximum flexibility | Requires self-management |

## Best Practices

<CardGroup cols={2}>
  <Card title="Use Access Tokens" icon="key">
    - Use registry-specific access tokens instead of passwords
    - Rotate tokens regularly
    - Grant minimal required permissions
    - Store tokens as Qovery secrets
  </Card>

  <Card title="Tag Strategy" icon="tag">
    - **Never use `latest`** in production
    - Use semantic versioning (v1.2.3)
    - Tag with Git commit SHA for traceability
    - Use different tags for different environments
  </Card>

  <Card title="Image Scanning" icon="shield">
    - Enable vulnerability scanning on your registry
    - Scan images before deployment
    - Use minimal base images (Alpine, Distroless)
    - Regularly update base images
  </Card>

  <Card title="Registry Reliability" icon="server">
    - Use registry close to deployment region
    - Enable image mirroring
    - Monitor registry rate limits
    - Have backup registry configured
  </Card>
</CardGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Authentication Failed">
    **Problem**: Cannot connect to registry or pull images

    **Solutions**:
    - Verify credentials are correct
    - Check registry URL format
    - Ensure registry is accessible from internet
    - For AWS ECR: Verify IAM permissions
    - For private registries: Check firewall rules
    - Token may have expired - regenerate
  </Accordion>

  <Accordion title="Image Not Found">
    **Problem**: Qovery cannot find specified image

    **Solutions**:
    - Verify image name is correct (case-sensitive)
    - Check tag exists in registry
    - Ensure image was pushed successfully
    - For private images: Verify registry is connected
    - Check image path format for your registry
  </Accordion>

  <Accordion title="Rate Limiting">
    **Problem**: Docker Hub rate limit errors

    **Solutions**:
    - Authenticate with Docker Hub account
    - Upgrade to Docker Hub Pro/Team
    - Use image mirroring/caching
    - Mirror frequently-used images to private registry
  </Accordion>

  <Accordion title="Slow Image Pulls">
    **Problem**: Deployments slow due to large image downloads

    **Solutions**:
    - Use registry in same region as cluster
    - Enable image caching
    - Optimize Docker images (multi-stage builds)
    - Use smaller base images
    - Remove unnecessary dependencies
  </Accordion>
</AccordionGroup>

## Security Considerations

### Registry Security

<Warning>
**Never commit registry credentials to Git**

Use Qovery's organization-level registry configuration instead of embedding credentials in code.
</Warning>

**Security Checklist**:
- ✅ Enable 2FA on registry accounts
- ✅ Use access tokens with minimal permissions
- ✅ Regularly scan images for vulnerabilities
- ✅ Use private registries for proprietary code
- ✅ Enable registry access logs
- ✅ Rotate credentials periodically
- ✅ Use HTTPS for all registry connections

### Image Security

**Best Practices**:
1. **Use official base images** from trusted sources
2. **Pin specific versions** - avoid `latest` tag
3. **Scan for vulnerabilities** before deployment
4. **Sign images** to verify authenticity
5. **Minimize attack surface** - remove unnecessary tools
6. **Run as non-root user** in containers
7. **Keep images updated** with security patches
