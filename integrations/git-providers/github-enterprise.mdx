## Overview

Qovery supports GitHub Enterprise Server, allowing you to connect your self-hosted or cloud-hosted GitHub Enterprise instance to deploy applications directly from your private repositories.

## Features

- **GitHub Enterprise Server**: Self-hosted GitHub instances
- **GitHub Enterprise Cloud**: Cloud-hosted enterprise plan
- **Organization Repositories**: Full access to enterprise organization repos
- **Personal Repositories**: Access to user repositories
- **GitHub Actions Integration**: Use GitHub Actions for CI/CD
- **Pull Request Environments**: Automatic preview environments
- **GitHub Container Registry**: Enterprise container registry support
- **Advanced Security**: GitHub Advanced Security features integration
- **SAML/SSO**: Support for enterprise authentication

## Prerequisites

<Check>GitHub Enterprise Server 3.0+ or GitHub Enterprise Cloud</Check>
<Check>Admin access to create OAuth apps in your GitHub Enterprise instance</Check>
<Check>Network connectivity from Qovery to your GitHub Enterprise instance (for self-hosted)</Check>
<Check>Valid SSL certificate (for self-hosted instances)</Check>

## Setup Instructions

### For GitHub Enterprise Cloud

<Steps>
  <Step title="Navigate to Organization Settings">
    In Qovery Console: Organization Settings → Git Providers
  </Step>

  <Step title="Select GitHub Enterprise">
    Click **Add Git Provider** → **GitHub Enterprise**
  </Step>

  <Step title="Choose Cloud">
    Select **GitHub Enterprise Cloud**
  </Step>

  <Step title="Authorize">
    Complete OAuth flow with your enterprise account
  </Step>

  <Step title="Grant Access">
    Authorize Qovery to access your enterprise organizations and repositories
  </Step>

  <Step title="Complete Setup">
    Repositories are now available for deployment
  </Step>
</Steps>

### For GitHub Enterprise Server (Self-Hosted)

<Steps>
  <Step title="Create OAuth App in GitHub Enterprise">
    In your GitHub Enterprise instance:
    - Go to **Settings** → **Developer settings** → **OAuth Apps**
    - Click **New OAuth App**

    Configure:
    - **Application name**: Qovery
    - **Homepage URL**: `https://console.qovery.com`
    - **Authorization callback URL**: `https://api.qovery.com/auth/github-enterprise/callback`
  </Step>

  <Step title="Get OAuth Credentials">
    After creating the app:
    - Copy **Client ID**
    - Generate and copy **Client Secret**
  </Step>

  <Step title="Add Provider in Qovery">
    In Qovery Console: Organization Settings → Git Providers → Add Provider

    Enter:
    - **Provider Type**: GitHub Enterprise Server
    - **GitHub Enterprise URL**: Your instance URL (e.g., `https://github.your-company.com`)
    - **Client ID**: From step 2
    - **Client Secret**: From step 2
  </Step>

  <Step title="Test Connection">
    Qovery will verify connectivity to your instance
  </Step>

  <Step title="Authorize">
    Complete OAuth flow through your GitHub Enterprise instance
  </Step>

  <Step title="Grant Access">
    Authorize Qovery to access repositories
  </Step>
</Steps>

## Network Requirements

For GitHub Enterprise Server (self-hosted):

### Firewall Rules

Allow outbound HTTPS traffic from Qovery to your GitHub Enterprise instance:
- **Protocol**: HTTPS (443)
- **Destination**: Your GitHub Enterprise Server
- **Source**: Qovery API servers

### Webhooks

Allow inbound HTTPS traffic from your GitHub Enterprise Server to Qovery:
- **Protocol**: HTTPS (443)
- **Destination**: `api.qovery.com`
- **Source**: Your GitHub Enterprise Server
- **Purpose**: Git push webhooks for auto-deploy

### DNS Resolution

Ensure your GitHub Enterprise Server hostname is resolvable:
- Public DNS records (recommended)
- Or configure Qovery to use your internal DNS

## Configuration

### Repository Access

Once connected, you can:
- **Deploy from any repository** in your enterprise organizations
- **Create preview environments** for pull requests
- **Auto-deploy** on push to specific branches
- **Use private repositories** without additional configuration

### Branch Protection

Qovery respects GitHub Enterprise branch protection rules:
- Protected branches can still be deployed
- Deployment status checks integrate with branch protection
- Required reviews don't block Qovery deployments

### Webhooks

Qovery automatically configures webhooks for:
- Push events (for auto-deploy)
- Pull request events (for preview environments)
- Tag events (for release deployments)

<Note>
Qovery uses organization-level webhooks when possible for better performance.
</Note>

## Advanced Configuration

### Custom SSL Certificates

For self-hosted instances with custom CA certificates:

1. Contact Qovery support to configure custom CA trust
2. Provide your CA certificate chain
3. Qovery will configure SSL verification

<Warning>
Self-signed certificates are not recommended for production use.
</Warning>

### IP Whitelisting

If your GitHub Enterprise instance requires IP whitelisting:

Contact Qovery support for the current list of IP addresses to whitelist.

### Proxy Configuration

For instances behind corporate proxies:

1. Ensure proxy allows HTTPS traffic to Qovery API
2. Configure proxy to pass through webhook requests
3. Test connectivity before connecting to Qovery

## GitHub Actions Integration

Use GitHub Enterprise with GitHub Actions CI/CD:

<Tabs>
  <Tab title="Build & Deploy">
    ```yaml
    name: Deploy to Qovery
    on:
      push:
        branches: [main]

    jobs:
      deploy:
        runs-on: self-hosted  # Use enterprise runner
        steps:
          - uses: actions/checkout@v3

          - name: Install Qovery CLI
            run: curl -s https://get.qovery.com | bash

          - name: Deploy to Qovery
            env:
              QOVERY_CLI_ACCESS_TOKEN: ${{ secrets.QOVERY_TOKEN }}
            run: |
              qovery application deploy \
                --application-name my-app \
                --commit-id ${{ github.sha }}
    ```
  </Tab>

  <Tab title="Build Image">
    ```yaml
    name: Build and Push to Registry
    on:
      push:
        branches: [main]

    jobs:
      build:
        runs-on: self-hosted
        steps:
          - uses: actions/checkout@v3

          - name: Log in to Registry
            uses: docker/login-action@v2
            with:
              registry: ghcr.io
              username: ${{ github.actor }}
              password: ${{ secrets.GITHUB_TOKEN }}

          - name: Build and Push
            uses: docker/build-push-action@v4
            with:
              context: .
              push: true
              tags: ghcr.io/${{ github.repository }}:${{ github.sha }}

          - name: Deploy to Qovery
            env:
              QOVERY_CLI_ACCESS_TOKEN: ${{ secrets.QOVERY_TOKEN }}
            run: |
              qovery application deploy \
                --application-name my-app \
                --commit-id ${{ github.sha }}
    ```
  </Tab>
</Tabs>

## Troubleshooting

<AccordionGroup>
  <Accordion title="Cannot Connect to GitHub Enterprise Server">
    **Problem**: Connection fails when adding provider

    **Solutions**:
    - Verify GitHub Enterprise URL is correct
    - Check firewall rules allow HTTPS from Qovery
    - Ensure SSL certificate is valid
    - Verify OAuth app credentials are correct
    - Check DNS resolution from Qovery
    - Test HTTPS access: `curl https://your-github-enterprise.com`
  </Accordion>

  <Accordion title="Webhooks Not Working">
    **Problem**: Auto-deploy not triggering on push

    **Solutions**:
    - Check webhook configuration in repository settings
    - Verify firewall allows outbound HTTPS from GitHub Enterprise
    - Ensure webhook URL is accessible: `https://api.qovery.com`
    - Check webhook delivery logs in GitHub Enterprise
    - Verify Qovery application has auto-deploy enabled
  </Accordion>

  <Accordion title="Repository Access Denied">
    **Problem**: Cannot access repositories after connecting

    **Solutions**:
    - Re-authorize Qovery OAuth application
    - Check organization permissions in GitHub Enterprise
    - Verify user has access to repositories
    - Check OAuth app has required scopes (repo, read:org)
    - Contact GitHub Enterprise admin to verify app approval
  </Accordion>

  <Accordion title="SSL Certificate Errors">
    **Problem**: SSL verification failing for self-hosted instance

    **Solutions**:
    - Verify SSL certificate is valid and not expired
    - Ensure certificate chain is complete
    - Check certificate matches hostname
    - For custom CA: Contact Qovery support
    - Temporarily test with IP address (not recommended for production)
  </Accordion>

  <Accordion title="Preview Environments Not Creating">
    **Problem**: Pull request environments not being created

    **Solutions**:
    - Verify auto-preview is enabled in Qovery
    - Check webhook includes pull_request events
    - Ensure branch protection doesn't block webhooks
    - Verify user permissions for creating environments
    - Check Qovery application logs for errors
  </Accordion>
</AccordionGroup>

## Enterprise Features

### SAML/SSO Integration

GitHub Enterprise SSO is transparent to Qovery:
- Users authenticate via enterprise SSO
- Qovery uses OAuth tokens post-authentication
- No additional Qovery configuration needed

### Advanced Security

Qovery integrates with GitHub Advanced Security:
- **Secret Scanning**: Prevents secret commits
- **Code Scanning**: Security analysis integration
- **Dependency Review**: Automated dependency checks
- **Security Advisories**: Vulnerability alerts

### Audit Logging

GitHub Enterprise audit logs track:
- Qovery OAuth app authentication
- Repository access by Qovery
- Webhook deliveries
- API requests from Qovery

### Compliance

Qovery on GitHub Enterprise supports:
- **SOC 2**: Compliance requirements
- **GDPR**: Data privacy regulations
- **HIPAA**: Healthcare data protection
- **FedRAMP**: Government compliance (with appropriate GHE setup)
